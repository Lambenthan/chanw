## 一、项目制：一个文件夹就是这篇论文的全部世界

网页版的 Claude 和 Claude Code 是两种东西。打开浏览器进 claude.ai，跟它说“帮我改一下论文第三章的措辞”，你得手动复制一段贴过去，它改完你再粘回 Word。这整个过程里它对你的整篇论文没有概念，看到的只有你贴过去的那一段。

Claude Code 绑定的是一个文件夹。你在终端 `cd` 进某个目录、敲下 `claude`，这个文件夹就成了它的工作台。目录里的论文 docx、do 文件、参考文献 bib、图表，它全部能读、能改、能跨文件对照。这就是项目制的含义。一篇论文从开题报告的第一行字到交稿那一刻，版本、数据、文献、图、引用全部沉淀在同一个文件夹里，这个文件夹就是这篇论文的全部世界。

## 二、启动那一秒，Claude Code 读了什么

敲下 `claude` 回车，它没有等你说话，先做了五件事。按官方文档的说法，加载顺序是先读 Anthropic 内置的系统提示，这部分改不了；再读当前环境，包括你在哪个目录、git 处于什么状态、文件夹里有哪些文件；接着从当前目录向上递归查找 CLAUDE.md，沿途各个层级全量叠加加载；然后读 `memory/MEMORY.md` 的前 200 行，或者前 25KB，谁先到取谁；最后读 `.claude/settings.json` 里你配的权限和钩子。

![Claude Code 启动时按顺序加载的五项：系统提示、当前环境、CLAUDE.md、MEMORY.md、settings.json，CLAUDE.md 与 MEMORY.md 是你能写入的两层](/images/essays/claude-code-empirical-workflow-02-startup@2x.png)

这一套读完大约占七八千个 token，相当于上下文窗口的 4% 左右。等于说你还没开口，它已经知道你在哪个项目、研究方向是什么、有哪些术语不能动、你之前对它说过哪些偏好。这一篇后面要做的，就是把 CLAUDE.md 和 MEMORY.md 这两层从无到有建出来，让下次启动它能读到。

动手之前还有两样配套工具值得先装。一样是语音输入工具，后面几条喂给 Claude Code 的 prompt 都不短，全程手敲很费劲，按一下快捷键说话、实时转成文字落到光标位置会省下不少时间。另一样是 git。写论文必须挂 git，原因是 AI 改文章会改坏，而且你当时不一定发现。我自己踩过一次，让它润色一段文献综述，它悄悄把一个方括号引用 `[17]` 删了，交上去导师审稿才看出来。项目下挂着 git，`git diff` 一秒钟就能比对出来。后面 CLAUDE.md 里那条“改 do 文件之前先 git commit 一次”的规矩，没有 git 根本立不起来。

## 第一步 · 从一坨乱文件到整齐的项目结构

把演示项目这个文件夹拖进 IDE，左侧文件浏览器里 95 个文件全部躺在根目录，没有任何子目录。这是研究生开新课题时桌面最常见的样子：二十多篇下载好的 PDF 文献，读过几篇没读的也下了；十几个 CSMAR 下载和外购的 dta 数据混在一起；十几个华证 ESG、绿色转型、管理者短视主义系列的 xlsx；几个之前写的 do 处理代码；一批各 dta 的字段说明 txt；还有没解压的 CSMAR 原始 zip 包。导师让你按这个开始写论文，多半无从下手。

接下来这一步我不手动整理，让 Claude Code 来整理。启动它之后，第一句话把项目背景和任务交代清楚：读一下当前目录所有文件，说明这是一个研究耐心资本对企业 ESG 表现影响的新论文项目，文件夹里有 CSMAR 下载的财务、治理、ESG 数据，有外购的管理者短视主义指标且分年报版和 MD&A 版两套，有文献 PDF、Stata do 文件和一堆字段说明 txt；要它做两件事，先把所有文件按用途分组列个清单，再建议一个合理的项目目录结构；最后加一句关键约束，先不要动文件，等确认了再执行。

这条指令有几处是有意设计的。项目背景那一句决定它分组时的判断口径；管理者短视主义有两套版本这种细节不说清楚，它分起来会乱；把任务拆成先列清单、再建议结构两步分开做，比一口气甩给它要稳；而“先不要动文件，等我确认”这条来自上一篇提过的“先确认方案、再动手”原则，是反复试错才总结出来的。

启动后 Claude Code 没有立刻动手。它先调了一个论文写作纪律 skill，名叫 `paper-claude-md-bootstrap`，这条纪律规定进新项目的第一件事是检查有没有 CLAUDE.md、没有就先停下来建。它把次序理清楚：先看清目录里有什么、列分组、给结构建议，等真要动手写论文前再补上 CLAUDE.md，然后才往下走。

扫目录时它先读到根目录里我之前写好的几份索引，`_按研究主题索引.md`、`_数据字典.md`、`测算方法说明.md`，分组就直接对齐我已有的分类：文献按 A1 到 D 几套测算口径分，数据按财务、治理、ESG 分。分类口径是我定的，它按这个口径把文件归位。

它扫完所有文件会给一份分组清单，核心原始数据、外购数据、ESG 评级、绿色转型、文献、代码、字段说明各归一类，再建议一套目录，大致是 `01_文献/`、`02_变量字典/`、`03_原始数据/`、`04_中间数据/`、`05_分析代码/`，其中 `03_原始数据/` 下面再按财务、治理、ESG 等分子目录。确认结构没问题，再回它一句具体执行要求：在当前目录建这些子目录，把对应文件用 `mv` 移过去、不留副本；文献 PDF 按我索引里的 A1 到 D 测算口径分进子文件夹；操作完给一份日志，每个文件移到了哪里都列出来；判断不了归属的文件单独列出来问，不许瞎放。它会建目录、批量移动、给出操作日志，几分钟之内，95 个文件散在根目录的局面就变成结构整齐的项目目录。

批量移动那 95 个文件之前它还做了一步：先 `git status` 确认工作区干净、记下 95 个受控文件，移完再用 git 核对一个没丢。批量改动前先确认基线、留下可回滚的痕迹，也是写进纪律的一条。

![在 IDE 里打开整理后的研究项目：左侧是 01_文献 / 02_变量字典 / 03_原始数据 / 04_中间数据 / 05_分析代码 的目录树，根目录放 CLAUDE.md、MEMORY.md、checkpoint.md 三份常驻文档；右侧是 CLAUDE.md 的内容](/images/essays/cc02-project@2x.png)

## 第二步 · 建 CLAUDE.md，写一份项目长期说明书

目录整齐了，但 Claude Code 还不知道这个项目该做什么、不该做什么，这要靠 CLAUDE.md。它是官方机制，放在项目目录下，每次会话开始全量加载进上下文。它分几个层级，用户级在 `~/.claude/CLAUDE.md`、项目级在项目根、本地级是 `CLAUDE.local.md`。写论文用的是项目级，跟着 git 走，换电脑、跟同门合作都拿得到。官方建议单份控制在 200 行以内，因为是全量加载，写太多白占 token。

让它生成 CLAUDE.md 时，我会把研究背景、要包含的几块内容、以及“先给草稿、确认后再写入文件”一并交代。研究背景写明课题、样本区间、被解释变量、核心解释变量和中介变量的口径。要它包含的内容里，要紧的一项是术语锁定表，列出在任何修改任务中都不得替换为同义词的术语。我自己实际遇到过，让 AI 润色一段方法论，它把“被解释变量”改成了“因变量”，整篇论文前后口径不一致，导师审稿一眼看穿。所以核心研究术语、计量术语、数据源名称、算法名称、写作人称这几类都得进锁定表，写作人称那一条要明确只用「本文」、不许换成「我们」「作者」「我」。

术语之外，CLAUDE.md 里还要固化几样长期不变的规矩。输出格式上规定学术中文、数字三位逗号分隔、百分比保留两位、改任何内容必须给修改前后对比、不写排比、不用空洞修饰词。项目目录约定上规定原始数据只读、处理后数据改前先备份、改 do 文件前先 git commit、回归日志写进 `runs/` 目录。导师固定要求里只放长期性的，标准误一律按 firm 聚类、中介效应用 Bootstrap 不接受 Sobel、稳健性至少跑替换核心解释变量加替换被解释变量加剔除金融危机年份子样本、结果必须有经济意义解读。再加上实证表格规范、变量命名约定、中文经管论文写作硬约束这几块，最后补一节“什么时候更新 CLAUDE.md”：新增核心术语进锁定表、导师明确了长期要求进固定要求、发现 AI 老犯某个错进禁做清单，临时性反馈则写到下一节的 MEMORY.md，不写进本文件。

上次开会随口说的“以后结论用被动语态”这种临时反馈就不写进固定要求，它属于 MEMORY.md 管。让它先出草稿、过一遍再回“可以，写入 CLAUDE.md”，它就落盘了。演示项目里我先把投稿目标和导师偏好留成占位，等这些定了再回填，CLAUDE.md 第一版里一共留了六处这样的待指定项。

## 第三步 · 建 MEMORY.md 与 checkpoint.md

CLAUDE.md 装的是不变的东西，那导师上次开会随口提的“以后结论部分要克制，别用‘显著证明’这种词”放哪儿，放 MEMORY.md。这是官方给的记忆机制，路径固定在 `~/.claude/projects/<项目对应 id>/memory/` 下，入口文件叫 MEMORY.md，启动时自动读前 200 行或 25KB。它的范围是项目级加机器本地，和 CLAUDE.md 有个关键差别，CLAUDE.md 跟 git 走，MEMORY.md 不跟，换电脑就没了。

官方只规定了路径和加载机制，怎么写、怎么组织完全没规定。我自己摸出来的一套是按功能分成 user、feedback、project、reference 四类，每个具体记忆文件用 frontmatter 标 type 方便检索，MEMORY.md 当索引、每条一行指向具体文件。让 Claude Code 建的时候，把这套结构和要先写入的条目一并交代：在项目根建一份 MEMORY.md 当跟 git 走的备份副本，另建 `memory/` 子目录按四类组织具体文件，user 装写作偏好、feedback 装导师和同行反馈、project 装项目阶段性信息、reference 装外部资源指针。再把已知的几条先写进去，比如导师反馈的克制用词、投稿期刊的参考文献格式、例会定的研究方向、主分析数据路径。有两类信息要明确排除，长期不变的硬约束已经在 CLAUDE.md 里了、MEMORY 别再来一遍，进度类信息归下面要建的 checkpoint.md 管、也不写 MEMORY。

往后加新记忆很简单，对话里直接说“记住：导师某日开会说结论要补一段政策含义、至少 200 字”，它就写进去并更新索引；或者敲 `/memory` 这个 slash 命令，它会列出当前所有 memory 文件让你挑一个编辑。

![MEMORY.md 是项目记忆的索引：每条一行指向 memory/ 下的具体记忆文件，按 user / feedback / project / reference 四类组织](/images/essays/cc02-memory@2x.png)

最后一份是 checkpoint.md，它和前两个不一样。它不是 Claude Code 的官方机制，Claude Code 不会自动读它，它是我加的一条写作约定，需要你通过指令让 AI 维护。加这一条的原因是写论文动辄几个月，会话一定会断，要么关电脑、要么撞上 rate limit，上下文一断 AI 就不知道你昨天写到哪了，我需要一份工作日记躺在文件夹里、下次重开能续上。它和 MEMORY.md 有大约三成功能重叠，但分工在三处不同：生命周期上 MEMORY 几周到几月才变一次、checkpoint 几天就改一次；加载上 MEMORY 启动自动读、checkpoint 得你说“读一下”才读；详细程度上 MEMORY 一句话概括、checkpoint 详细到段落级。划分原则是会跟着写作进度变的状态只写 checkpoint，“在写 3.4 节”“卡在固定效应那段”进 checkpoint，“标准误用 cluster”“投稿目标某刊”这种几个月不变的进 MEMORY 或 CLAUDE.md。

立这条约定时让 Claude Code 做两件事。一是建初始的 checkpoint.md，分已完成、进行中、待做、待解决四块，初始内容写“项目刚启动、目录已整理、CLAUDE.md 与 MEMORY.md 已建好、下一步开始写第三章方法论”这种状态。二是把维护规则写进 CLAUDE.md 的写作流程那一节，规定每完成一节、跑完一组回归、整理完一份文献清单就更新 checkpoint.md，每次重开如果用户说“先读一下 checkpoint.md”就优先读它、从断点接着做。把维护规则写进 CLAUDE.md，以后每次启动 AI 都自动读到这条约定，不用你每次重复。

![checkpoint.md 是一份工作日记：分已完成、进行中、待做、待解决四块，重开会话说一句“先读一下 checkpoint.md”就能从断点接着做](/images/essays/cc02-checkpoint@2x.png)

至于 Claude Code 真正官方的会话保存，是另外两个东西。`claude --continue`，简写 `-c`，恢复上一次被打断的对话连同历史；对话中按两次 Esc 回退到上一个检查点、撤销最近的改动。这两个管的是对话本身，checkpoint.md 管的是写作进度，两条线并行用。

![三份常驻文档的六维对照表：CLAUDE.md 装项目长期信息且跟 git 走，MEMORY.md 装阶段性记忆且机器本地，checkpoint.md 装当下进度由 AI 按指令维护](/images/essays/claude-code-empirical-workflow-02-three-files@2x.png)

## 第四步 · 重启验证三份文档生效

三份文档建完，重启 Claude Code 验证它们真的生效了。终端里 `exit` 退出当前会话，重新敲 `claude` 启动。第一句话问它知不知道这个项目是做什么的，列一下从 CLAUDE.md 和 MEMORY.md 读到的关键信息，再检查 checkpoint.md 说明现在在哪个阶段。

![真实会话截图：重启后让 Claude Code 复述项目进度，它读完 checkpoint 后说清研究方向，还列出耐心资本、ESG、机制变量等几处尚未定的口径决策](/images/essays/cc02-session-verify@2x.png)

如果它能正确复述出研究方向是耐心资本对 ESG 的影响、有若干条术语锁定、导师有几条固定要求、当前阶段在写第三章方法论，三份文档就全部生效了。下次再开 Claude Code，它就有了正确的起点，不必再从零交代背景。整个过程你几乎没动手敲文件，目录、CLAUDE.md、MEMORY.md、checkpoint.md 全是 Claude Code 写的，你只是把每一步该做什么、要包含什么内容用人话告诉它。这就是 Claude Code 加项目制加常驻文档的搭法。

## 收尾 · 把这套设定接回实证流程

这一篇把项目地基打好了。回到上一篇那个原则：实证的终点是一份能跑通的分析文件，我会让 Claude Code 和 Stata 同时跑同一份任务、并要求 Stata 输出 do 文件，两端结果对一遍。项目制和常驻文档正好支撑这种交叉校验。CLAUDE.md 里“改 do 文件前先 git commit”“标准误一律按 firm 聚类”这些规矩落在文件里，Stata 端和 Claude Code 端跑的就是同一套口径，对出来的差异才有意义；术语锁定表保证两端对变量的称呼一致，不会一边叫被解释变量、一边叫因变量。

把演示项目里那套流程原样搬到你自己的论文目录上，从 `cd` 进目录、`claude` 启动，到让它建三份文档，把“耐心资本”换成你的研究主题、把术语锁定表换成你的术语，其余照搬即可。建完重启问它一句“你知道这个项目是做什么的吗”，能正确复述，项目制的基础就搭好了。下一篇从文献调研与阅读开始。
